# alibre-neutralizer — Code Review (Correctness)

**Date:** 2026-06-20
**Scope:** Second-opinion review, code only (correctness bugs). Reviewed both Python exporters (`alibre-neutralizer.py` and `alibre-neutralizer-addon/src/Scripts/alibre-neutralizer.py`), the C# IronPython host (`AlibreAddOn.cs`), the add-on descriptor/project files (`.adc`, `.csproj`), and the example XML config. The two Python files are identical except the add-on copy logs to a WinForms `OutputConsole` instead of `print`, so Python findings below apply to both unless noted.

**Summary: 8 bugs — 3 High, 3 Medium, 2 Low**

## High
- **`alibre-neutralizer.py:630-635`** (and add-on copy `:668-673`) — Missing `return`/abort after the "no config selected" branch. When the user cancels the Open dialog, the code shows an `ErrorDialog` but then falls straight through to `AlibreNeutralizer(CurrentAssembly(), cfg_file_path)` with `cfg_file_path` equal to `""` or `None`. `ET.parse("")`/`ET.parse(None)` then raises, producing a confusing crash instead of a clean exit. The error message claims "Alibre Neutralizer will close now," but control flow does not actually stop.
- **`alibre-neutralizer.py:399-410`** (`_export_subassemblies_recursive`, and add-on copy `:437-448`) — Subassembly export (Step 2) has no dedup guard, unlike parts (Step 1) and the recursive descent (Step 3). The top-level driver loop `export_all` (`:349-352`) also calls the recursive function for every entry in `root_component.SubAssemblies` without checking `processed_files` first. Real assemblies routinely contain multiple instances of the same subassembly file; each instance is re-exported (and, because both resolve to the same `{Number}_{Name}` path, repeatedly re-walked/re-written). The whole module's stated goal — "we only export each component once" — is violated for subassemblies. The fix is to wrap Step 2 in the same `if subassembly.FileName not in already_processed_files` check used elsewhere and to guard the `export_all` loop.
- **`alibre-neutralizer.py:511-512`** (`_export`, and add-on copy `:549-550`) — Error-swallowing on export failure. The `except Exception` catches every export failure (ExportSTEP*/ExportSAT/ExportIGES/ExportSTL/CSV) and only prints a message, discarding `e`. A failed export therefore looks like success: `export_all` completes, `main` shows "The export process completed!", and the component is still marked processed so it is never retried. For a bulk-export tool, silently producing an incomplete output set is a correctness defect. At minimum the exception detail should be logged and the failure surfaced to the final summary.

## Medium
- **`AlibreAddOn.cs:76-84`** — `IAlibreAddOn.LoadData`/`SaveData` (the overloads taking `global::AlibreAddOn.IStream`, which is the interface's actual signature — note the file aliases the unrelated `System.Runtime.InteropServices.ComTypes.IStream` at line 11) `throw new NotImplementedException()`. The no-op overloads at lines 71-72 use the *aliased* COM `IStream` and so do not satisfy the interface. If Alibre invokes `LoadData` on add-on load (it can, independent of `HasPersistentDataToSave`, which only gates `SaveData`), the add-on throws. Safer to make these no-ops like the intended stubs.
- **`alibre-neutralizer.py:295 / 333`** (`getattr(ExportTypes, directive.find('type').text)`) — Unvalidated config-driven attribute lookup. A misspelled or unsupported `<type>` (e.g. `STEP242`) returns whatever attribute exists or raises `AttributeError`; worse, a name that happens to match a real attribute (e.g. `get_file_extensions`) would be accepted as an "export type" and fail far downstream. The surrounding `convert_to_string` (`:43-59`) also has no final `else`, so an out-of-range type silently returns `None`, which is then `.format`-ed into log lines. There is no whitelist check that the parsed value is one of the seven defined export-type integers.
- **`alibre-neutralizer.py:439-481` + `105-120`** — `AssembledSubAssembly` is treated as if it exposes assembly geometry/structure, but per the AlibreScript API (and the bundled stub `AlibreScript.py`, where `AssembledSubAssembly` exposes only `Name`/`Configurations`/`GetConfiguration`/`GetSelectionAssembly`) it is the *placement* wrapper, not an `Assembly`. The recursion calls `subassembly.Parts`, `subassembly.SubAssemblies`, and `subassembly.FileName` (`:394-404`) and `_export` calls `ExportSTEP*`/`ExportSAT` directly on it (`:483-510`). If those members/methods are not present on `AssembledSubAssembly`, every subassembly export and the recursion throws — and the throw is swallowed by the High bug above, so subassemblies silently never export while parts do. This should be verified against the live API; if confirmed, the subassembly path must convert via `GetSelectionAssembly()` (or equivalent) before accessing `Parts`/`SubAssemblies`/`FileName`/`Export*`.

## Low
- **`alibre-neutralizer.py:161-162 / 536-537`** — The path-sanitizer character class `[^\w_.: \-\\]` preserves `:`. After `os.path.normpath`, a property value containing `:` (e.g. a `Description` like `Rev: A`) survives into a path segment; on Windows a `:` inside a filename is illegal (interpreted as an alternate-data-stream separator), so the eventual `ExportXXX`/`open` can fail. `:` is allowed so drive letters survive, but it is not constrained to position 1, so it can leak into filenames.
- **`alibre-neutralizer.py:412-437`** (`_purge_according_to_export_directive`) — `os.walk(purge_path)` over a non-existent directory yields nothing (fine), but `endswith(file_extension)` matches case-sensitively. Purge extensions are lowercase (`.stp`, `.step`), while Alibre/Windows files frequently carry uppercase extensions (`.STP`). Old uppercase exports would not be purged, leaving stale files alongside freshly written ones — partially defeating the purge feature. (Export and overwrite still work, hence Low.)

## Notes on items checked and deemed NOT bugs
- No integer-division hazards: the code performs no `/` arithmetic; the only numbers are the `ExportTypes` enum-ish constants used purely for `==` comparison, so IronPython 2.7 true-vs-floor division is irrelevant here.
- CSV files are correctly opened in binary mode (`open(..., 'wb')`) with `csv.writer`, which is the right pattern to avoid blank interleaved rows under Python 2.7 on Windows. The `with` block ensures the handle closes.
- `get_prettified_component_properties` correctly uses `getattr(component, key, None)` and treats `None`/`""` as "Undefined …"; the comment about `not component_value` swallowing `0`/`False` does not apply because the actual condition used is the safe `is None or == ""`.
- Purge ordering in `export_all` is correct: all purges (Step 1) run before any export (Steps 2–4), so a directive cannot delete files another directive just wrote.
- `_export` creates the destination directory tree (`os.makedirs`) before writing; the `os.path.exists` guard avoids the race-free-enough double-create in this single-threaded tool.
- The C# `ScriptRunner` reuses one `ScriptEngine`/`ScriptScope` and re-`SetVariable`s the session each run; this is single-threaded (UI-invoked), so no concurrency/deadlock concern, and there is no obvious COM-object leak (the script obtains `Root` itself and Alibre owns lifetime).
- `_bool_from_elem` parsing of the enable flags is correct, including the default-True-when-missing behavior matching the documented config contract.
